# 개발 정책 (Development Policy)

> **수립일:** 2026-04-15
> **목적:** 개발 속도와 문서 품질을 함께 유지하기 위한 최소 규칙
> **적용 대상:** 모든 커밋 (사람이 직접 하든 AI 에이전트를 통해 하든)

---

## 원칙

> **"코드가 바뀌면 문서도 바뀐다."**
>
> 문서가 코드보다 뒤처지면, 다음 개발자(미래의 나 포함)는 잘못된 지도를 보고 길을 간다.
> 배포된 코드와 문서가 어긋나면 그 프로젝트는 이미 기술 부채를 갖고 있는 것이다.

---

## 1. 커밋 전 필수 체크리스트

커밋하기 **전에** 아래 항목을 확인한다.

### 1-1. 공통 (모든 커밋)
- [ ] 변경 내용이 **`CHANGELOG.md`** 에 반영되었는가?
  - 새 기능 / 버그 수정 / 보안 / 리팩토링 / 문서 변경 중 어디에 해당하는지 분류
  - 배포일 기준 섹션에 한 줄 이상 기록
- [ ] 커밋 메시지에 **Phase 번호** 또는 **분류 태그**가 포함되었는가?
  - 예: `Phase 13: ...`, `fix: ...`, `docs: ...`, `refactor: ...`
- [ ] `.env` / 비밀값 / API 키 / 개인정보가 포함되지 않았는가?

### 1-2. 기능 추가 / 변경
- [ ] **`PRD.md`** — 사용자 스토리·기능 리스트에 새 기능이 반영되었는가?
- [ ] **`TRD.md`** — 스키마·API 스펙·아키텍처 변경이 반영되었는가?
- [ ] **`DEVPLAN.md`** — 해당 Phase 체크박스가 업데이트되었는가?
- [ ] **`context.md`** — 현재 상태 요약이 최신인가?

### 1-3. DB 마이그레이션
- [ ] `backend/src/migrations/NNN_name.sql` 파일이 추가되었는가?
- [ ] 마이그레이션이 **로컬과 배포 DB 모두에 적용**되었는가?
- [ ] `TRD.md` 의 DB 스키마 섹션이 업데이트되었는가?

### 1-4. API 변경
- [ ] `TRD.md` API 스펙 테이블이 최신인가?
- [ ] Breaking change인 경우 `CHANGELOG.md` 에 명시했는가?

### 1-5. 보안 관련 변경
- [ ] `TRD.md` 의 보안 섹션이 업데이트되었는가?
- [ ] `CHANGELOG.md` 의 Security 섹션에 기록되었는가?

### 1-6. 배포 설정 변경 (환경변수, Railway/Vercel 설정)
- [ ] `TRD.md` 의 배포 섹션이 업데이트되었는가?
- [ ] 필요 시 `.env.example` 이 동기화되었는가?

## 2. 문서별 책임 범위

| 문서 | 언제 업데이트? | 한 줄 요약 |
|---|---|---|
| `PRD.md` | 사용자에게 보이는 기능이 바뀔 때 | "무엇을 만드는가" |
| `TRD.md` | 기술 스펙(스키마/API/스택/보안)이 바뀔 때 | "어떻게 구현하는가" |
| `DEVPLAN.md` | Phase 단위 작업을 시작/완료할 때 | "언제 무엇을 하는가" |
| `CHANGELOG.md` | **모든 커밋** — 한 줄이라도 | "배포별 변경 이력" |
| `context.md` | 중요 상태 변경 시 (URL, Phase 완료 등) | "빠른 개요" |
| `DEV_POLICY.md` | 정책이 바뀔 때만 | "어떻게 일하는가" (이 파일) |

## 3. 커밋 메시지 컨벤션

```
<타입>: <한 줄 요약>

<본문 — 변경 이유, 영향 범위>

Co-Authored-By: ... (AI 에이전트 사용 시)
```

### 타입 태그

| 태그 | 용도 |
|---|---|
| `Phase N:` | 계획된 Phase 작업 |
| `feat:` | 새 기능 추가 |
| `fix:` | 버그 수정 |
| `refactor:` | 기능 변경 없는 리팩토링 |
| `docs:` | 문서만 수정 |
| `security:` | 보안 개선 |
| `chore:` | 빌드 설정, 패키지 업데이트 등 |
| `test:` | 테스트 추가/수정 |

---

## 4. Phase 작업 흐름 (권장)

```
1. DEVPLAN.md 에서 Phase 범위 확인
         ↓
2. 필요 시 PRD/TRD 업데이트 (설계 단계)
         ↓
3. 코드 작성
         ↓
4. 로컬 검증 (curl/브라우저 테스트)
         ↓
5. DEVPLAN.md 체크박스 ✅ 표시
         ↓
6. CHANGELOG.md 에 변경 요약 추가
         ↓
7. context.md 최신화 (상태가 크게 바뀌었다면)
         ↓
8. git commit + push  ← 이 시점에 모든 문서가 최신이어야 함
         ↓
9. 배포본 검증 (health check, 실제 기능)
```

## 5. 예외 규정

### 급한 핫픽스 (production down 등)
- 코드만 먼저 배포 **허용**
- 단, **같은 날 안에** 문서를 소급 업데이트
- 커밋 메시지에 `hotfix:` 태그 사용
- 후속 커밋 메시지 예: `docs: hotfix abc123 내용 CHANGELOG 반영`

### 실험 / 탐색 작업
- 별도 브랜치에서는 문서 업데이트 생략 가능
- main 머지 시점에는 정책 100% 준수

## 6. AI 에이전트 개발 시 추가 규칙

이 프로젝트는 Claude Code 등 AI 에이전트로 개발됩니다.
AI가 커밋을 수행할 때는 다음을 **반드시** 따릅니다.

1. 커밋 직전에 변경 파일 목록을 스캔하고, 필요한 문서 업데이트를 자동으로 수행
2. 문서 업데이트 없이 커밋 진행 시, 사용자에게 명시적으로 보고하고 확인 요청
3. 커밋 메시지 하단에 `Co-Authored-By` 추가
4. 배포 후에는 `CHANGELOG.md` 의 해당 항목을 반드시 작성

## 7. 정책 위반 시

- 다음 커밋에서 즉시 보완 커밋(`docs:` 태그) 생성
- 반복 위반 시 이 정책 문서 자체를 재검토
